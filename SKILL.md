# 深度调研技能

## 模式选择

启动时确认两个参数：

| 参数 | 选项 | 默认 |
|------|------|------|
| 协作模式 | **单 Agent** / **多智能体** | 单 Agent |
| 交付形式 | **轻量**（口头讨论） / **文档**（飞书文档+可视化） | 轻量 |

用户未指定时用默认值，但主动询问确认。

## 调研流程

### 定义

明确调研边界：
- 调研对象是什么（产品/技术/框架/工具）
- 调研目的（评估是否采用、竞品分析、学习借鉴、了解动态、技术选型等）

深度原则：**调研到信息来源允许的最深处。** 能拿到源码就看源码，能拿到一手文档就看文档，只有公开报道就把公开报道吃透。不预设上限，但也不强求不存在的信息。

### 调研

**由单个 Agent 完成全部信息收集与深度分析。** 派出一个 agent，携带 [调研维度清单](references/dimensions.md) 作为 prompt 的一部分，一次性完成从全貌扫描到深度拆解的全部工作。

Agent prompt 必须包含：
- 调研对象和目的
- 完整的维度清单内容（直接嵌入，不要让 agent 自己去读文件）
- 竞品数量约束：**核心竞品不超过 2 家**，深度对比；其余一句话带过

Agent 在单次执行中自行决定搜索节奏：先获取全貌，再对核心能力深挖。不需要分阶段、不需要多个 agent。

维度对照规则：
- 逐一检查清单中的每个维度
- **适用的** → 必须覆盖
- **不适用的** → 给出跳过原因，并提供等价替代（如"架构图"不适用时，可能"因果链路图"是等价替代）
- **用户有权要求补上被跳过的维度**，调研者必须执行

清单是下限不是上限。调研者应根据具体对象自主判断是否需要额外维度。

#### 核心能力

核心/特色能力需要深度展开，且**必须以完整段落形式阐述，而非要点罗列**。每项核心能力应写成连贯的说明性段落：先讲它是什么、解决什么问题，再讲它的运作逻辑、策略和关键规则，最后讲它为什么有效。深度以"读者能通过阅读段落理解其运作原理并判断其价值"为标准。

写作规范：
- **逻辑和策略的阐述用段落**：一个功能的工作机制、设计理念、技术原理，必须写成有因果关系的完整段落，让读者能跟随思路理解。不要用碎片化的要点列表代替逻辑阐述
- **对比和特点列举用表格/要点**：当需要多维度对比、功能清单枚举时，表格和要点列表是合适的形式
- 判断标准：如果内容需要读者"理解为什么"，用段落；如果内容只需要读者"知道有什么"，用表格/要点

信息来源不足时诚实标注，不假装已经写深了。

#### 一般能力

一句话定义 + 有什么即可，不展开。

#### 竞品对比

竞品对比是标准动作，必须包含。核心竞品**不超过 2 家**（最直接的竞争对手），深度对比优势、劣势、适用场景差异。其余相关产品一句话带过即可，用户追问时再展开。

#### 信息获取原则

优先参考官方文档等一手来源，同时鼓励多渠道获取信息（社区讨论、用户评测、技术博客等），以获得更立体的视角。信息来源不足时诚实标注。

### 审计

**此阶段是强制的，不可跳过。在用户追问之前完成。**

自我反思，检查：
- 对照 [调研维度清单](references/dimensions.md)，适用维度是否都覆盖了
- 信息是否有事实性错误
- 核心能力是否写到了逻辑和策略层面，而非停留在标题级描述
- 是否有明显的遗漏领域

发现问题立即补全，不等用户指出。

#### 多智能体模式附加

在自查之上，质疑者介入：
- 质疑者可自行获取补充信息来验证或补全调研结论
- 产出两类问题：
  - **补全型**：调研者遗漏的事实、能力、模块
  - **深挖型**：调研者提到但未充分展开的关键点
- 调研者回应质疑，补全信息

### 呈现

向用户展示调研成果：
- 经过审计补全的完整调研结论
- [多智能体] 质疑者的问题清单（展示给用户）

用户可以：
- 直接追问自己关心的方向
- 挑选质疑者的问题继续展开
- 认为已经足够，进入下一阶段

### 用户追问

基于已经完整的信息进行深入讨论。此时用户的追问质量更高——从"你漏了什么"变成"我对这个点有不同看法"或"这个机制具体怎么运作"。

每个追问都要回答到"最深"——不存在"差不多就行了"。

### 提炼

从信息收集上升到判断性观点。提炼的输出应该能直接支撑决策，不是信息的再次罗列。

### 交付

#### 轻量模式
口头总结关键结论，不生成文档。

#### 文档模式
生成飞书文档（调用 lark-doc skill），包含：
- 完整调研内容（按维度组织）
- **可视化图表**插入文档对应位置（见下方图表规范）
- 用户追问的问题 + 完整回答（阶段 6 的全部讨论内容，不遗漏）
- 提炼的判断性观点
- 竞品对比表

##### 图表交付规范

**禁止用 ASCII art 或代码块伪装图表。** 所有架构图、流程图、线框图必须渲染为真正的可视化图表，通过飞书画板插入文档。

具体流程：
1. **选择渲染路径**：分层架构图用 DSL（JSON），流程图/时序图/思维导图用 Mermaid（.mmd）
2. **渲染为 PNG**：使用 `@larksuite/whiteboard-cli` 渲染，视觉审查确认无截断、无重叠
3. **插入文档**：在飞书文档对应位置创建空白画板（`<whiteboard type="blank"/>`），从响应中获取 `board_token`，然后通过 `whiteboard-cli --to openapi` 上传到画板

调研 Agent 产出的文本描述（如 ASCII 架构图）是**中间产物**，写入飞书文档时必须转换为可视化图表，不能直接以代码块形式粘贴。

## 调研维度

必选维度清单维护在 [references/dimensions.md](references/dimensions.md)。

调研者在调研和审计时必须对照此清单。清单是下限不是上限——覆盖清单之外，调研者仍可自由发挥。

用户可随时修改此清单，skill 自动适应。

## 关键约束

1. **深度不妥协**：始终调研到信息来源允许的最深处，不接受表面了解
2. **核心能力用段落阐述**：核心能力的逻辑、策略、机制必须写成完整的说明性段落，不能用要点列表代替逻辑阐述。表格和要点仅用于对比和枚举场景
3. **审计不可跳过**：审计是强制的，在用户追问之前完成
4. **维度必须对照**：清单里的每一项都要逐一对照，适用的必须覆盖
5. **诚实标注不确定性**：信息来源不足时必须标注，不假装已经写深了
6. **判断优于罗列**：调研的终点是可以支撑决策的认知，不是信息堆砌
7. **中文交付**：所有输出使用中文，包括架构图标注
8. **单 Agent 调研**：信息收集由单个 agent 一次性完成，不拆分为多个 agent，避免重复搜索
9. **竞品上限**：核心竞品不超过 2 家，其余一句话带过
10. **图表必须可视化**：文档模式下，架构图/流程图/线框图禁止用代码块或 ASCII art 代替，必须通过 whiteboard-cli 渲染为真正的可视化图表并上传飞书画板。调研 Agent 的文本图表是中间产物，不是最终交付物
