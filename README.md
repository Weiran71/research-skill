# research-skill

> 面向产品经理的 AI 深度产品调研技能 for Claude Code

## 这不是"帮你搜一搜、总结一下"的工具

市面上不缺 AI 调研助手，但它们给你的多半是信息的罗列——功能清单、优缺点列表、竞品表格。**research-skill** 做的是产品解析：当你面对一款产品，需要理解它的设计哲学、拆解核心机制、看透竞品差异、提炼可迁移的产品认知时，这个技能才是对的工具。

它的调研维度来自一位有丰富 AI 产品经验的产品经理的实战沉淀——不是"这个产品有什么功能"，而是"它为什么这样设计""核心机制怎么运作""对我做产品有什么启发"。每一条维度都指向产品决策，而非信息堆砌。

## 核心能力

### 产品经理视角的调研维度

不是泛泛的"优缺点分析"，而是面向产品决策的 6 大维度：

1. **整体架构与流程** — 产品的骨架长什么样，配架构图
2. **设计理念与思想** — 为什么这样设计，而不是别的方式
3. **核心/特色能力详解** — 独有的东西怎么运作、为什么有效
4. **一般能力简述** — 有什么，一句话带过
5. **竞品对比** — 真正的差异化，不是功能列表对比
6. **洞察与启发** — 对你做产品、做 demo、做决策有什么可迁移的认知

维度清单由用户维护（`references/dimensions.md`），你可以根据自己的产品视角增删调整，技能自动适应。

### 深度不妥协

- **核心能力用段落阐述**：拒绝用要点列表代替逻辑阐述。每项核心能力写成完整的说明性段落——先讲是什么，再讲怎么运作，最后讲为什么有效
- **调研到信息来源允许的最深处**：能拿到源码就看源码，能拿到一手文档就看文档，只有公开报道就把公开报道吃透
- **诚实标注不确定性**：信息不足时标注，不假装写深了

### 强制自查审计

调研完成后自动执行审计，在你追问之前就把遗漏补上：
- 对照维度清单逐一检查覆盖度
- 核心能力是否写到了逻辑和策略层面
- 是否有事实性错误或明显遗漏

### 双模式协作

| 模式 | 说明 | 适用场景 |
|------|------|---------|
| **单 Agent** | 单调研者 + 自查审计 | 快速了解一款产品 |
| **多智能体** | Researcher + Challenger 对抗审查 | 严肃的竞品分析、产品选型 |

多智能体模式需配合 [agent-research-team](https://github.com/Weiran71/agent-research-team) 插件使用。

### 双档位交付

| 交付形式 | 说明 |
|---------|------|
| **轻量** | 口头讨论，不生成文档 |
| **文档** | 飞书文档 + 可视化图表（通过飞书画板渲染，非 ASCII art） |

## 它产出的不是报告，是认知

传统调研的终点是"信息汇总"。**research-skill** 的终点是**判断性观点**——直接支撑你做产品决策的认知，不是信息的再次罗列。

例如，调研扣子 2.5 后，它不会只告诉你"扣子有云电脑功能"，而是告诉你：

> "Agent 竞争正在从'智力竞赛'转向'环境竞赛'——当模型能力趋同时，决定 Agent 价值的不再是谁更聪明，而是谁拥有更完整的执行环境。"

这才是能指导你下一步产品决策的认知。

## 安装

将此目录放入 Claude Code 的 skills 目录：

```bash
# 方式一：直接克隆到 skills 目录
git clone https://github.com/Weiran71/research-skill.git ~/.claude/skills/research-skill

# 方式二：已有项目，创建符号链接
ln -s /path/to/research-skill ~/.claude/skills/research-skill
```

## 使用

在 Claude Code 对话中直接说：

```
帮我调研一下 Cursor 这款产品
```

技能会自动触发，询问你选择协作模式和交付形式后启动调研。

## 触发词

`调研` `深度调研` `分析` `研究` `了解`

## 设计原则

- **深度不妥协** — 调研到信息来源允许的最深处
- **核心能力用段落阐述** — 拒绝要点列表代替逻辑阐述
- **审计不可跳过** — 在用户追问之前完成自查
- **维度必须对照** — 清单逐一检查
- **诚实标注不确定性** — 信息不足时标注，不假装写深了
- **判断优于罗列** — 终点是支撑决策的认知，不是信息堆砌

## 进阶：自定义调研维度

编辑 `references/dimensions.md`，增删你关心的维度。技能会自动适应你的清单。

例如，如果你更关注商业模式分析，可以加入：

```markdown
### 7. 商业模式与变现逻辑
- 收入模型是什么
- 付费转化的关键设计
- 定价策略背后的逻辑
```
